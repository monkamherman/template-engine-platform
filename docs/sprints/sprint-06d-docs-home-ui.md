# Sprint 06D — Documentation Home UI

## Goal

Implement the documentation home page `/{locale}/docs` as a vertically finished UI surface.

This sprint translates the approved Stitch direction into clean Next.js, Tailwind and shadcn/ui-compatible components. Do not copy the Stitch HTML.

## Branch

```text
feat/sprint-06d-docs-home-ui
```

Create this branch from the latest `main` after the documentation PR that introduces this sprint has been merged.

## Required reading

1. `AGENTS.md`
2. `docs/product-scope.md`
3. `docs/interface-inventory.md`
4. `docs/design-prompts/README.md`
5. `docs/design-prompts/00-global-stitch-guidelines.md`
6. `docs/design-prompts/05-docs-home.md`
7. `docs/design-reviews/05-docs-home-stitch-review.md`
8. This sprint file.

## Scope

Implement the page content for:

```text
/{locale}/docs
```

The implementation must preserve the existing routing, locale handling and application shell.

## Non-negotiable rules

Do not:

- modify or duplicate the global header;
- modify or duplicate the global footer;
- add a separate docs top navbar generated by Stitch;
- copy Stitch HTML directly;
- use Tailwind CDN;
- use Google Fonts external links;
- use Material Symbols;
- use inline CSS;
- use inline JavaScript;
- hardcode fake version numbers;
- invent compatibility guarantees;
- invent support levels not approved by the product model;
- scatter raw hex values inside components.

Use:

- existing project layout and route conventions;
- shadcn/ui-compatible primitives;
- `lucide-react` icons;
- central Tailwind/shadcn color tokens;
- localized FR/EN content structure.

## Required sections

### 1. Breadcrumb

Inside the page content, show a light breadcrumb:

```text
Home / Docs
```

Use the project locale route helpers if available.

### 2. Hero

Include:

- badge: `Documentation`;
- headline: install, activate and operate the WooCommerce Template Engine with confidence;
- short body copy explaining self-installation, guided setup, official access, store model configuration, updates and troubleshooting;
- prominent search/command visual.

The search can be a non-connected visual preview if search is not yet implemented, but it must not fake working persistence.

### 3. Quick actions

Add quick links:

- Start here;
- Installation;
- Activation;
- Troubleshooting.

### 4. Path selector

Create three cards:

1. Starter self-install
   - Requirements;
   - Installation;
   - Activation;
   - Initial setup.

2. Pro guided launch
   - Getting started;
   - Onboarding checklist;
   - Staging validation;
   - Launch preparation.

3. Managed operations
   - Updates;
   - Backup and rollback;
   - Release notes;
   - Support.

Keep the copy careful: these are paths and support boundaries, not guarantees.

### 5. Documentation categories

Create a category grid for:

- Getting started;
- Requirements;
- Installation;
- Activation and licenses;
- Initial setup;
- Shop models;
- Niches;
- Localization;
- Products;
- Payments;
- Customization;
- Updates;
- Backup and rollback;
- Troubleshooting;
- Release notes.

Each card should include:

- icon;
- title;
- short description;
- one to three link chips or badges;
- optional status badge such as Essential, Setup, Operations or Support.

### 6. Highlighted guides

Add three highlighted guides:

- Install from a clean WordPress environment;
- Activate official access safely;
- Prepare backup and rollback before updates.

### 7. Support bridge

Add a support bridge with three cards:

- Check troubleshooting;
- Open account support request;
- Compare Pro and Managed support.

Do not promise dedicated channels unless approved by product copy.

### 8. Review notice

Use a subtle `Alert`:

```text
Documentation is versioned. Compatibility, legal and support claims must match the approved release and review state.
```

Make it professional, not alarming.

### 9. Version/release card

Add a small card for:

- Current docs version;
- Compatible theme version;
- Last updated placeholder or approved value;
- Release notes link.

Use placeholders only when real release data is not connected. Make placeholders visibly non-final if shown.

## Color system

## Component guidance

Prefer extracting reusable page-level components where it improves clarity:

```text
DocsHero
DocsQuickActions
DocsPathSelector
DocsCategoryGrid
DocsCategoryCard
HighlightedGuides
DocsSupportBridge
DocsVersionCard
DocsReviewNotice
```

Avoid over-abstraction if the current repository convention favors route-local sections.

Use available primitives:

- Button;
- Card;
- Badge;
- Alert;
- Separator;
- Breadcrumb or breadcrumb-like markup;
- Command or command-style search visual;
- Input;
- lucide-react icons.

## Localization

Preserve FR/EN support from day one.

Do not hardcode English-only strings directly in a way that prevents localized page content.

## Responsive requirements

Desktop:

- premium documentation landing page;
- strong search presence;
- path selector and category grid easy to scan.

Tablet:

- two-column sections where readable;
- no fixed sidebars.

Mobile:

- search first;
- quick actions immediately visible;
- path cards stack;
- categories remain readable;
- no horizontal overflow.

## Tests and checks

Run:

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

If any command fails, document the reason and keep the implementation PR in draft.

## PR requirements

The implementation PR must include:

- summary of sections delivered;
- components created or changed;
- screenshots or notes for desktop/tablet/mobile preview;
- proof that header/footer were not duplicated;
- explanation of the color token usage;
- list of Stitch claims removed or softened;
- tests executed;
- remaining limitations.

## Acceptance criteria

This sprint is complete only when:

- `/fr/docs` and `/en/docs` render correctly;
- the page uses existing shell chrome;
- no generated header/footer is present inside the page;
- the page uses project tokens, not raw Stitch colors;
- no Tailwind CDN, Google Fonts, Material Symbols, inline CSS or inline JS is introduced;
- docs search visual is accessible even if not connected;
- Starter, Pro and Managed paths are clear;
- installation, activation, troubleshooting and releases are prominent;
- the layout is intentionally responsive;
- all required checks are run or documented.
